Line 1015 is the test's own `with_defence(id, 1) - with_defence(id, 400)`

## Wiring that needs files outside this lane's path set

Three concrete wiring opportunities were identified but not taken, because each
needs a file another lane owns. They are listed with the exact edit so whoever
picks them up does not have to re-derive it.

## 1. `StFrame` does not carry the sector header's frame dimensions

`crates/mdec/src/st_ring.rs`. `StRing::deliver_sector` parses the STR sector
header and keeps `frame_number` and `frame_size_bytes`, dropping the `+0x10`
width and `+0x12` height. Retail's frame poll (`FUN_801CF740`) uses exactly
those two to re-programme the decode rects every frame -
`legaia_mdec::str_player::DecodeEnv::apply_frame_dimensions` is the port, and it
is inert only because `StrPlayer::next_frame` has nothing to pass it.

The edit is two `u16` fields on `Slot` and `StFrame`, populated in the same
place `frame_size_bytes` is, then one call in `next_frame`. That turns a
disclosed-inert port into a live one and makes the engine's decode geometry
follow the bitstream the way retail's does.

## 2. `World::battle_intro` carries no per-style state

`crates/engine-core/src/world/encounters.rs`. `World::tick_battle_intro` drives
`battle_intro_transition::tick_transition` (the phase counter) and nothing else.
The five style emitters are now ported -
`legaia_engine_vm::battle_intro_styles` / `battle_intro_tiles` /
`battle_intro_swirl`, plus the two seeders in `battle_intro_particles` - and all
are inert for the same reason: no host owns their working set.

Wiring them is two steps, and the second is the real cost:

1. `World` gains a per-encounter style working set (an enum over the four
   allocated shapes) seeded on the frame the session enters `Transition`, and
   `tick_battle_intro` ticks it alongside the phase counter.
2. `legaia-engine-render` gains a transition pass. All five styles texture a
   **captured field framebuffer**, which the engine does not produce - it
   presents through a swapchain rather than a re-readable VRAM. This is the
   blocker, not the state plumbing.

Step 1 alone is worth doing only if step 2 is planned; a ticked field nothing
draws is an inert call with a cost.

## 3. `docs/reference/functions.md` was left alone

Eleven function entries were newly decoded this lane (`FUN_801CFDA0`,
`FUN_801D0370`, `FUN_801D081C`, `FUN_801D0D24`, `FUN_801D0E54`, `FUN_801D11D0`,
`FUN_801D1564`, `FUN_801D1888`, `FUN_801D1A20`, `FUN_801D27E0`,
`FUN_801D5E20`). All eleven are documented in
`docs/subsystems/cutscene.md`; the shared `functions.md` directory was not
touched to avoid a merge conflict with other lanes. One row per address, citing
`overlay_field_battle_intro_<addr>.txt` /
`overlay_cutscene_dialogue_<addr>.txt`, would close that gap.
