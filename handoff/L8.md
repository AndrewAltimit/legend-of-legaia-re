# Handoff from lane L8 (pause menu / save screen / shop / equipment / inventory)

## 1. Misrouted coordinator instruction: the HP-bar drain belongs to the battle lane

Mid-run the coordinator sent L8 a disassembly-verified brief on the battle
HP-bar accumulator `actor[+0x10]` and asked for it to be ported "for your Task
C". L8's Task C is `docs/subsystems/save-screen.md`; the HP-bar work lands in
`crates/engine-vm/src/battle_action/**` and `crates/engine-core/src/battle_session/**`,
neither of which is in L8's declared path set. L8 did **not** edit those files -
touching them would collide with whichever lane owns the battle-action slice.
The facts are reproduced verbatim below so nothing is lost.

### The invariant

`(+0x172 displayed) - (+0x14C live) == (+0x10 accumulator)`. Battle-action
state `0x51` parks forever when a party slot has `+0x14C != +0x172` while
`+0x10 == 0`, because the drain is gated on a non-zero accumulator.

### `FUN_80047430` - the drain (`ghidra/scripts/funcs/80047430.txt`)

- Guard at `0x800474E8`: `lw a0,0x10(s2); beq a0,zero,<skip>` - a zero
  accumulator means the bar is not touched at all.
- **Party slot (`slot < 3`)**: quarter-step. `step = (acc+3)>>2` for `acc > 0`,
  `acc>>2` for `acc <= 0`; the `+3` / `-3` bias is what keeps `step` non-zero
  so the drain always terminates. Then `0x172 -= step` **and** `acc -= step`
  with the *same* step, so total bar movement equals the seeded accumulator
  exactly, for either sign. Writeback at `0x80047574`.
- **Non-party slot (`slot >= 3`)**: one shot at `0x80047578` - `bar -= acc;
  acc = 0` in a single frame. It reads the accumulator with `lhu` (low halfword
  only), so a negative accumulator on a monster wraps. That is a retail quirk;
  a faithful port keeps it.
- MP runs the same shape with `+0x174` displayed / `+0x178` accumulator.
  **Retail bug worth a `// REF:` note**: the non-party MP arm at `0x80047624`
  operates on `+0x172` / `+0x10` (the *HP* fields), not `+0x174` / `+0x178`, so
  a monster's MP accumulator is never cleared and its displayed MP never
  updates. Copy-paste bug in retail; port as-is and say so.

### Seeding - two conventions, and the asymmetry is the softlock's root cause

- `FUN_801EC3E4` (damage / heal kernel) **accumulates** at all four sites:
  `0x801EDAF0` (`acc -= (max-hp)`, overheal branch where live HP saturates at
  `+0x14E` first), `0x801EDB14` (`acc -= net`), `0x801EDB58` (`acc += net`,
  second actor), `0x801EDB7C` (`acc = bar` anti-overkill clamp, guarded by
  `if (bar < acc)` at `0x801EDB70`). Read-modify-write everywhere, so
  overlapping hits compose and the invariant survives.
- `FUN_800402F4` (item / restore applier) **assigns**: `hp = hp + amount` at
  `0x800408AC`, then a bare `sw v0,0x10(v1)` with `v0 = -amount` at all three of
  `0x800408FC`, `0x80040D28`, `0x800410BC`. The old accumulator is never read,
  so a restore landing mid-drain discards the remainder and the bar settles
  exactly that remainder away from live HP with `acc == 0`.

### `FUN_801E7250` - the settle gate

Dispatches on the *acting* actor's `+0x1DD`: `0..2` compares that party actor's
`+0x14C` vs `+0x172`; `3..7` returns 0 immediately; `8` scans actor-table slots
`0 .. ctx[+0x00]-1`, and `ctx[+0x00]` is the **party member count** (measured as
3 in the Gaza 2 capture), so the all-target arm scans the party side only;
`> 8` returns 0.

### `FUN_801E752C`

Force-assigns `+0x172 = +0x14C` after each status HP write (`0x801E7600`,
`0x801E7698`) - the only re-sync in the corpus.

### The test the coordinator most wants

Alongside "enters and exits `0x51` on settle": seed a damage accumulator, step
the drain partway, apply a restore through the *assigning* path, then assert the
bar settles off live HP with `acc == 0` and that the `0x51` gate never releases.
That is the engine-side reproduction of the community bug.

The coordinator states `docs/subsystems/battle-action.md` § "Where the desync
comes from: two seeding conventions" already carries all of this in the main
checkout, and asked that no lane edit that file.

## 1b. The `casting.rs` debug-only failure: test bug, not a kernel defect

The coordinator assigned L8 the pre-existing panic at
`crates/engine-core/src/world/battle/casting.rs:1015`
(`capture_bypass_tests::the_bypass_wrapper_weights_the_defence_terms_more_heavily`,
`attempt to subtract with overflow`, debug profile only). **Verdict: the damage
kernel is correct and the test's premise was wrong.** No capture-class kernel
was changed.

Measured curve in the test's own fixture (`power = 2000`), damage vs the
defender's `battle_defense[0]`:

| defence | bypass `0x5C` | respect `0x5B` |
|---|---|---|
| 1 | 524 | 1002 |
| 50 | 528 | 999 |
| 100 | 528 | 996 |
| 200 | 528 | 990 |
| 400 | 528 | 977 |
| 800 | 528 | 952 |

The bypass hit is flat, so `dmg(1) - dmg(400)` underflows `u16`. The cause is
`spell_wrapper_predamage`'s **bonus arm**: it re-rolls the attacker whenever
`atk < def + power`, and the re-roll is
`wrapper_bonus_roll(def, power, r) = def + power + r % ((power >> 2) + 1)`
(instruction-identical between `FUN_801DD4B0` and `FUN_801DD6B4`). The net
`atk - def` it leaves is `power + r % …` - the defender's contribution
**cancels exactly**. With a high-power move the arm fires at every defence
worth sampling, so a bypass hit is defence-insensitive end to end even though
`spell_defender_roll` really does weight `+0x15C` / `+0x160` at `>> 1`, eight
times the physical wrapper's `>> 4`. Nothing inverts, and nothing produces a
heal-shaped result: `before - hp` was positive at every sample.

Fix applied, in the test only:

- `the_bypass_wrapper_weights_the_defence_terms_more_heavily` now asserts the
  `>> 1` vs `>> 4` weighting on the pure kernels (`spell_defender_roll` vs
  `physical_defender_roll`), where it is unconditionally observable, instead of
  end to end where the bonus arm hides it.
- A new sibling, `the_bonus_arm_floor_makes_a_bypass_hit_defence_insensitive`,
  asserts the real end-to-end property over five defence samples: the bypass
  hit is constant and positive, the respecting hit is monotonically
  non-increasing and strictly lower at the top of the span. All arithmetic is
  `i32`, so an inversion is now an assertion failure in **both** profiles
  rather than a debug-only panic and a silent release pass.

Worth noting for the wave: the failure mode here - `u16` subtraction of two
measured quantities inside a test - is invisible under `--release`. Other
battle tests that difference two damage samples should be swept for the same
shape.

## 2. L8 findings other lanes may want

### A one-site wire for whoever owns the field VM's take-item opcode

`engine-core::equipment::party_unequip_accessory_by_id` (`FUN_800430AC`) has a
pinned retail caller, and it is not a menu flow. In `FUN_801DE840` at
`0x801E1ABC..0x801E1AEC` - read off the **disassembly**, since the field-VM
dump's decompiled C for this arm is not backed by disassembly in the same file
(`overlay_0897_801de840.txt` shows the C at its line 6287 but no matching
instructions; the instructions are in `overlay_world_map_801de840.txt` line
3235 onward and its siblings):

```
jal  FUN_8004313C            ; install the active item window
lbu  a0, 1(s6)               ; operand = item id
jal  FUN_80042310            ; consume 1 of `id` from the bag
 li  a1, 1
sll/sra v0, 16               ; sign-extend the i16 result
li   v1, 0x100
bne  v0, v1, <continue>      ; found in the bag -> done
lbu  a0, 1(s6)
jal  FUN_800430AC            ; NOT in the bag -> strip it party-wide
 addiu s8, s8, 3             ; delay slot: pc += 3 on both paths
```

So the take-item opcode falls back to the party-wide accessory unequip when the
bag does not hold the item - a script that takes away an accessory the player is
**wearing** still removes it. The engine's take-item host removes from the
engine inventory only and has no bag-miss branch, so the ported kernel is inert.
Wiring is one call site in the field VM's take-item opcode host. L8 sharpened
the `NOT WIRED:` reason in `crates/engine-core/src/equipment.rs` to name this
site; the sibling give path is `0x801E0448` (`FUN_8004313C` then
`FUN_800421D4(id, 1)`).

### Two port-catalog name collisions fixed by renaming, not by tooling changes

The catalog's call graph resolves calls by name with no receiver inference, so a
free function called `step` picks up an edge from every `x.step(...)` **and from
every parameter named `step`**, and a method called `word` picks up every
unrelated `word(...)` closure. Both shapes were making deliberately-inert ports
read as wired. L8 renamed the two in its own files
(`menu_open_sequence::step` -> `menu_open_step`, `EquipStatBlock::word` ->
`stat_word`) rather than touching `scripts/ci/port-catalog.py`. Other lanes
holding an anchor that the audit calls live for no visible reason should check
the anchor's *name* before assuming the wire exists.
