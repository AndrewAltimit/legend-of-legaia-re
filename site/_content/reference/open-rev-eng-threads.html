Open reverse-engineering threads

An index of reverse-engineering questions about Legaia's runtime, and where each one stands - open, answered, or already disproved. Rows are questions, not progress markers.

How to use this page

Before starting a hunt, look for it here. A question about this game is usually in one of three states, and this page tells you which: it is open and the row names the next step; it is answered and the row points at the doc or parser that owns the fact; or it has a tempting wrong answer that was already falsified, and the row records why.

That third case is the point of the page. Most rows here are closed, not open. A falsified row is kept forever with its reasoning intact, because the reasoning is the deliverable - "the world-map slot-4 bodies are coastline wireframes" is a very plausible reading of those bytes, and knowing why it is wrong is worth more than the row it occupies. The instructive dead ends are collected in the do-not-re-walk register.

Each row carries a one-line current answer; byte-level detail lives on the linked page, and the full working notes live in docs/reference/open-rev-eng-threads.md.

Status badges:

  • open (plain text) - active hunt; a concrete next step exists.
  • partial - main result pinned; a residual sub-question remains (see the details block below each table).
  • resolved - answered; the row cites the evidence and the page that now owns the fact.
  • falsified - hypothesis disproved; the row is kept so the path isn't re-walked.

Many statuses are qualified in parentheses - resolved (structural), resolved + ported, falsified (VA-aliasing artifact) - naming how the thread was settled. Read the parenthetical: resolved (static) and resolved (RAM-verified) are different strengths of evidence.

Debug flags - resolved

An exhaustive corpus sweep finds zero writers of 0x8007B8C2 and zero byte-granular references to 0x8007B98F - the latter because it is the most-significant byte (offset +3, little-endian) of the 32-bit debug-mode word 0x8007B98C, and code reads the word. _DAT_8007B98C is the debug gate tested by the input dispatcher FUN_8001822c plus ~14 resident field-overlay (0897) gates, written by the shared menu/title/save-init routine. Poking 0x8007B98F = 1 sets the MSB, so every _DAT_8007B98C != 0 gate reads debug active and SELECT+△ opens the debug menu. Statically pinned; no uncaptured-overlay reader needed.

PROT 0896 - resolved

Host-capstone decode of the head (extraction entry 896, located by the "FWIN ERR" bytes) identifies it as the vestigial Japanese-build field-menu / config / status overlay: a Shift-JIS config/menu/status label pool, the "FWIN ERR %d" (Field WINdow) debug printf at file offset 0x3D4, real MIPS at link base ~0x801D0000 indexing the 0x414-byte character records, and head function-pointer tables across 0x801D81C0..0x801DC700 - the same window-renderer VA family as the live English menu overlay PROT 0899, which ships the English equivalents of the identical label set with zero FWIN. A distinctive-signature scan across 140 catalogued RAM states finds 0896 resident in none (positive control: the English "Battle Voices" is resident in 10 menu-phase states), so the USA build never loads it - a vestigial JP-build overlay carried on the disc.

Slot-B cluster - residual

  • 0924 "Ultimate Rave" / 0925 are stager-shaped (arithmetic ids place them as other enemies' specials); one mid-cast capture each pins the binding. 0926 is a single-sector non-stager.
  • The 0x4000/0x4001 render-mode nodes are statically decoded (moving particle / 3D positional sound emitter) and the classification is ported, but no catalogued capture seats one live - an enemy cast of a Sim-Seru creature (Palma / Mule / Jedo) would.

Off-by-2 ripple - residual

Only the evolved-Seru legs 0x90 → 918 and 0x91 → 919 stay arithmetic-predicted (no mid-cast capture); every other leg of the player, evolved, high, and enemy stager blocks is capture-pinned on the linear spell − 0x79 arithmetic.

  2. The next step is non-obvious from the code or git log. If grep would surface it, no row needed.
