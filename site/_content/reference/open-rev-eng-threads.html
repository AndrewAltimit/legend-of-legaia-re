Open reverse-engineering threads

An index of still-open hunts and the negative findings worth not re-walking. Rows are questions, not progress markers - each states the current answer and links to the format or subsystem page holding the detail.

How to use this page

Use this page to find what's worth digging into next. Each row carries a one-line current answer; byte-level detail lives on the linked page, and the full working notes live in docs/reference/open-rev-eng-threads.md.

Status badges:

  • open (plain text) - active hunt; a concrete next step exists.
  • partial - main result pinned; a residual sub-question remains (see the details block below each table).
  • resolved / falsified - answered or disproved; the row is kept so the path isn't re-walked. Instructive dead ends are collected in the do-not-re-walk register.

Debug flags - resolved

An exhaustive corpus sweep finds zero writers of 0x8007B8C2 and zero byte-granular references to 0x8007B98F - the latter because it is the most-significant byte (offset +3, little-endian) of the 32-bit debug-mode word 0x8007B98C, and code reads the word. _DAT_8007B98C is the debug gate tested by the input dispatcher FUN_8001822c plus ~14 resident field-overlay (0897) gates, written by the shared menu/title/save-init routine. Poking 0x8007B98F = 1 sets the MSB, so every _DAT_8007B98C != 0 gate reads debug active and SELECT+△ opens the debug menu. Statically pinned; no uncaptured-overlay reader needed.

PROT 0896 - resolved

Host-capstone decode of the head (extraction entry 896, located by the "FWIN ERR" bytes) identifies it as the vestigial Japanese-build field-menu / config / status overlay: a Shift-JIS config/menu/status label pool, the "FWIN ERR %d" (Field WINdow) debug printf at file offset 0x3D4, real MIPS at link base ~0x801D0000 indexing the 0x414-byte character records, and head function-pointer tables across 0x801D81C0..0x801DC700 - the same window-renderer VA family as the live English menu overlay PROT 0899, which ships the English equivalents of the identical label set with zero FWIN. A distinctive-signature scan across 140 catalogued RAM states finds 0896 resident in none (positive control: the English "Battle Voices" is resident in 10 menu-phase states), so the USA build never loads it - a vestigial JP-build overlay carried on the disc.

Do-not-re-walk register
  • Slot-4 → cluster-A converter: none exists; the pool is filled only by FUN_80026B4C via type-2/9 packs.
  • "FUN_8001E54C distributes slot-4 records into a working buffer": falsified - the handlers read the payload in place.
  • World-map "coastline wireframe" / kingdom-border reading: falsified - the bodies are object-local 3D meshes.
  • Walk-view continent as per-cell instanced meshes, and the "single grass page, +0x14 unused" reading: both falsified - it is a heightfield and +0x14 IS the tile selector.
  • Navmesh at 0x80108EA4: GPU primitive scratch; region data is in the field-file preamble, not the field-pack schema slots.
  • "Summon magnitude in the FUN_801f2d68 jump table": falsified - that code is animation/GPU only; the magnitude is applied by the paired stager overlay.
  • FUN_801F7088 as the player-summon rotation path: falsified by live trace - the summon draws through the ordinary battle actor path (FUN_80048A08, TRS keyframes).
  • FUN_801F811C as a summon part transform: it is the 2D screen-mask (iris) widget of PROT 0900's widget family, exclusive to the ending scenes.
  • "PROT 1204 is the battle party mesh/palette source": falsified - battles assemble from the player files; 1204 is the Baka Fighter / default-equipment sibling.
  • Battle-palette hunts with the bit-15-set runtime needle: the disc source is bit-15-clear (the uploader ORs STP on) - why every raw brute missed.
  • "Rows 490..497 are the party palette band" / character CLUTs routed through 0x8007BEC0: falsified - scene-environment palettes; battle relocates TSB/CBA to rows 481-483.
  • Stat growth from a "Seru struct +0x74": falsified - it is the static DAT_80076918 parameter block.
  • Steal data in the PROT 867 monster record: not there - it is the separate static SCUS table DAT_80077828.
  • A per-scene encounter-record table on disc: none - encounters are inline dispatcher-op operands.
  • Placement-NPC text_id resolving through the scene MES: it never does - dialogue is inline 0x1F segments; the "prompt + options" and "box-geometry header" readings are falsified.
  • "Picker labels are the 2-byte entries": falsified - the entries are signed relative jumps; labels are ordinary 0x1F segments.
  • FUN_8001ebec as the dialog-box renderer: it is a per-character TMD pose copier; the dialog SM is FUN_80039b7c, the pager FUN_801D84D0.
  • Per-cue reverb opt-in ("Spirit/echo cues only"): falsified - Studio C is master-on globally with broad EON routing.
  • A title-screen entry in the mode table: no such entry - the title overlay loads pre-mode-dispatch.
  • Debug flag 0x8007B98F "stripped at link time / inert" AND "consumer in an uncaptured overlay": both falsified - it is the MSB of the debug word 0x8007B98C, whose readers (FUN_8001822c + field-overlay gates) are statically pinned.
  • PROT 0896 as the menu overlay, or as the mode-24 overlay at 0x801C5818: both refuted (menu = 0899; the base was an over-read artifact). It is the vestigial Japanese-build sibling of 0899, never loaded by the USA build.
  • "PROT 0905 = Gimard": the + 0x381 off-by-2 - Gimard 0x81 arithmetics to extraction 0903; the Nighto "Hell's Music" dance-song reading is refuted (it is the attack's display name).
  • XP table at 0x8007123C / 0x80070A3C: both wrong; the real table is DAT_80076AF4.
  • Menu-glyph atlas as boot-resident field VRAM: it is menu-time-resident; uploading it at field entry flags a wrong static texel.
  • Equipment table +4 as AGL/evasion: falsified - it is SPD.
  • "PROT 0028 = town0c's .MAP": misattribution - 0028 is izumi's; town0c's map (PROT 0019) is byte-identical to town01's.

When to add a row

A thread belongs here when:

  1. There is something specific that would close it - a probe to run, a dump to read, a function to port. "Generally understand X better" is not closable; skip.
  2. The next step is non-obvious from the code or git log. If grep would surface it, no row needed.
  3. The detail lives elsewhere (a docs page, a Ghidra dump). The row is the pointer, not the analysis.

When a thread closes, rewrite the row to a resolved / falsified line if the path was instructive enough to warrant a do-not-re-walk marker; otherwise delete the row. Rotating the page is part of using it.
