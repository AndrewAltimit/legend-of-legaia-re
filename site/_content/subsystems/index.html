Subsystems

How Legaia's engine actually works under the hood. Each card is a self-contained page on one subsystem — what it does, where its code lives, and how it connects to the rest. Start with Boot path if you're new to the project.

The shape of the engine

Legaia is a fairly typical late-90s PSX RPG engine, but with a twist: most of the gameplay logic doesn't live in the main executable. SCUS_942.54 contains the boot path, low-level I/O, the asset dispatcher, the move-table VM, the motion VM, and the audio / renderer libraries — about half the runtime. The other half lives in RAM overlays that the disc loads on demand: title screen, town, battle, menu. The two highest-level scripting VMs (the field/event VM and the effect VM) live in those overlays.

The list below groups subsystems by what layer of the engine they sit at.

Bootstrap and asset plumbing

Boot path
What runs before the title screen. Loads the PROT.DAT TOC into RAM, populates the asset-type dispatcher, and hands control to the 28-mode game-state machine.
Asset loader
How per-scene asset bundles are stitched together at runtime. Battle, field/town, sound bank, and BGM-lookup loaders each follow the same dev/retail split.

The four script VMs

Legaia's runtime is driven by four independent bytecode VMs that all talk to one shared actor model. They serve different layers and were almost certainly written by different members of the dev team.

Actor / sprite VM
A small fixed-width VM (13 opcodes) driving the title screen's animated sprites. Lives in the title-screen overlay. Fully ported.
Move-table VM
Per-actor animation, motion, and state. 71 opcodes in the main exe plus 61 sub-opcodes in the town overlay extension. Drives Tactical Arts.
Motion VM
Per-actor pursue / patrol / face-target VM at FUN_8003774C. Drives NPC pathing + camera follow + "face the speaker" dialog poses.
Field / event script VM
The big one: NPC movement, dialog triggers, cutscene sequencing, story flags. 43 opcodes plus a default-route fourth flag bank. Fully ported.

Per-domain runtime

Effect VM
Battle-spawn effects (spell casts, hit sparks, item-use animations). A per-slot state machine living in the battle overlay, fed by a 32-master / 128-child slot pool.
Cutscene (STR mode)
PSX STR video decode + XA audio playback. Game modes 26/27 (StrInit / StrMode). MDEC decoder: VLC → IDCT → BT.601 YCbCr→RGBA. Clean-room port in crates/mdec.
Battle
Battle scene loader, the per-action state machine, character record layout, the per-frame stat aggregator, range / line-of-sight, and the weapon-trail builder.
Battle action FSM
The 47-state machine inside FUN_801E295C — the layer between “player picked Attack” and “HP has been deducted.” Two-level dispatch: action category (party byte) plus execution phase (ctx byte).
Battle formulas
Damage / MP-cost / accuracy / RNG arithmetic. Selector dispatch in FUN_800402F4; spirit damage is hard-coded; MP cost is ability-bit modified; engine-vm port lives in battle_formulas.rs.
Audio
Sound-driver path strings, three SCUS consumers, the SsAPI sequencer cluster (statically linked PsyQ libsnd), and the libspu DMA engine that moves bytes into SPU RAM.
Renderer
The Legaia TMD renderer (60 GTE ops with a per-mode descriptor table). The engine port emulates PSX VRAM in a compute-friendly format so multi-page meshes render correctly.
