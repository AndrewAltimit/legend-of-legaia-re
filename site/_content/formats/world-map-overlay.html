World-map overlay outlines Inferred

The slot 4 payload of each world-map (kingdom) bundle holds the wireframe/outline data the dev-menu top-view renderer draws over the world map. Three carriers, one per kingdom: 0085 map01 (Drake), 0244 map02 (Sebucus), 0391 map03 (Karisto).

Confidence

Inferred — container format pinned, semantic confirmed by visualisation, RAM payload byte-verified. The outer pack + per-body header layout is exact across all 15 sub-bodies of Drake (16 each for Sebucus and Karisto). The decoded payload matches live RAM byte-for-byte (zero diffs over 32304 bytes). The per-body draw routine has not been traced in Ghidra yet, so the on-disc → draw-mode dispatch is still open.

Where this lives in the 7-asset bundle

The kingdom bundles use the standard 7-asset scene_asset_table shape with type sequence (1, 2, 3, 4, 5, 6, 7). Slot 4's type byte is 0x05 per asset-type — the standard table calls this "MOVE", but the kingdom-bundle consumer interprets the bytes as world-map outline data, not as a move table. This is the same kind of CDNAME / type-byte mismatch already documented for vab_01 and move_program_no: the type byte is just a routing tag, and the kingdom path uses it differently.

Layout

Outer pack

+0x00   u32  count                ; number of sub-bodies
+0x04   u32  byte_offsets[count]  ; absolute byte offset into the
                                  ; decoded payload (NOT word offsets,
                                  ; unlike the slot-1 TMD pack)
+offset bodies[count]             ; contiguous sub-bodies

Drake decodes to 32304 bytes with count = 15. First entry is 0x40 = 4 + 4*15 (right after the header). Each subsequent entry is strictly greater; the last body extends to end-of-decoded-payload.

Sub-body header (8 bytes)

+0x00   u8   count_a              ; records per group
+0x01   u8   flag_a               ; usually 0; 1 for kind=4 boundary frames
+0x02   u8   count_b              ; number of groups
+0x03   u8   flag_b               ; usually 0
+0x04   u16  marker               ; constant 0x080C across all bodies
+0x06   u16  kind                 ; 1, 2, or 4 (semantic ambiguous)

Body payload

+0x08   record[count_a * count_b] ; each record is 8 bytes
                                  ; ( i16 x, i16 y, i16 z, i16 attr )
+...    trailer (8 bytes)         ; always 8 zero bytes in Drake

Total body size is always 8 + count_a * count_b * 8 + 8. The math fits every body in all three kingdoms exactly.

Per-kingdom body inventory

Drake = 15 bodies; Sebucus = 16; Karisto = 16. The flag_a byte is 0 for all kind = 1 and most kind = 2 bodies and flips to 1 for the kind = 4 boundary-frame bodies (plus one anomalous kind = 2 body in Karisto). The leading three bodies (kind = 1, count_a = 10) are byte-identical templates across all three kingdoms — the same generic inner-contour shape installed at the front of each slot. The kingdom-specific data lives in the trailing bodies.

RAM layout

Slot 4 is loaded verbatim into RAM with zero per-byte diffs vs disc. Drake's payload starts at 0x8011A624 (the outer pack header) and ends at 0x80122454 exclusive — exactly 32304 bytes, matching the disc-decoded length. Body 0's records start at 0x8011A664 (0x40 past the base, after the 4-byte count and 15 × 4-byte offsets). No runtime fixup is applied.

Confirmed by scripts/pcsx-redux/diff_slot4_ram_vs_disc.py against a PCSX-Redux save state: every byte of all 15 bodies matches the disc-side LZS-decoded payload. The load base was pinned by signature-searching the full 2 MiB main RAM for the 64-byte outer pack header (count = 15 followed by byte_offsets[0..15]) — see scripts/pcsx-redux/autorun_dump_full_ram.lua for the procedure if the base needs re-pinning on a different build / save state.

Why this isn't the bulk continent terrain

The top-down rendering of all bodies' points reveals body 12 traces a continent coastline and body 13 traces the world boundary — both are 1D contours, not 2D terrain meshes. Total ~4000 vertices across all 15 sub-bodies is far below the ~17000+ vertex positions the textured continent in the GPU prim pool would need (Drake's pool has 4994 prims, mostly POLY_FT4 quads with 4 verts each).

The bulk continent geometry source is therefore separate. The most likely candidate is a procedural emitter, sibling of the horizon emitter FUN_801D7EA0 reachable from the per-frame world-map tick FUN_80016444. Pinning that is still open.

Consumer (presumed)

Slot 4 is loaded by the standard kingdom-bundle loader. The reader that interprets these bytes is presumably the world-map controller FUN_801E76D4 (top-view branch) and/or the developer menu renderer FUN_801EAD98, both documented under world map subsystem. The exact reading call has not been pinned down in Ghidra; the data does not appear as a direct LUI+ADDIU reference to 0x8011A624 in any captured world-map overlay, suggesting the consumer accesses the buffer via a runtime pointer rather than a hardcoded address.

Bodies 9 / 10 / 11 have Y spans of 18-27K - comparable to the X/Z scale, not the small elevation deltas a 2D map contour would carry. Body 12's Y span is only 4K (consistent with a near-flat coastline at near-zero elevation). When all bodies are projected onto xz (top-down) some of them produce coherent silhouettes (12) while others look like noise (9 / 11) - because for those bodies the flatten-to-xz step is collapsing genuine 3D structure into 2D mush.

Rendering the same bodies on xy (front side view) surfaces clean vertical pillar-like silhouettes for bodies 9 and 11 - 3D mesh structure that's invisible in the top-down view. This suggests slot 4 is heterogeneous: at least some bodies are object-local 3D meshes (collision hulls? instantiable decoration meshes?) rather than 2D map contours. The world-overview viewer exposes the projection plane as a second selector (xz / xy / zy) so the same data can be inspected from each angle without committing to one interpretation.

It's also possible the engine doesn't treat the middle field as world Y-elevation at all - some bodies make more sense as map data in the xy projection (cleaner overall continent shape) than in xz. The final word will come from Ghidra capture of the consumer.

Topology hypothesis

Inspecting Drake body 12 group 0 records by hand reveals records pair up at fixed X-bands — (r0, r1), (r2, r3), ... are 5 (lo, hi) X-pairs per group, with consecutive groups acting as differential Z-updates over a shared topology. The body is therefore a count_a × count_b grid of vertices forming a heightfield strip, not a chained polyline. Three rendering candidates are wired into the CLI and the web viewer:

  • pairs — each group emits count_a / 2 independent edge pairs ((r0,r1), (r2,r3), ...); 120 z-stride groups × 5 edges per group = 600 line segments;
  • grid — emits both row edges ((k, g) → (k + 1, g)) and column edges ((k, g) → (k, g + 1)); shows the heightfield as a quad-mesh wireframe;
  • points — topology-free; matches the visible structure most closely and is the recommended raw-validation lens.

None of row, col, pairs, or grid cleanly reproduces the dev-menu top-view's continent silhouette by itself, so the runtime consumer likely encodes more than one draw mode (probably keyed off kind / flag_a). The world overview viewer exposes all five rendering styles via the "show slot-4 wireframe" toggle.

Open questions

  1. kind = 1, 2, 4 semantic. Not yet tied to a draw-mode or sub-format. kind = 4 correlates strongly with flag_a = 1 across all three kingdoms (every kind = 4 body has flag_a = 1) and those bodies plot as world-boundary frames at ±32K / large-perimeter rings — but the reverse doesn't hold (Karisto body 10 is kind = 2 with flag_a = 1), so (kind, flag_a) together select more than a simple "border-strand" toggle.
  2. Per-record 4th int16 column (attr). Always 0 for body 4; 22 distinct values across 300 records in body 5; 214 distinct in body 12. Body 12 attr-values cluster at ±1280, ±1792, 1793, ±1281, ±1025 — which look like packed (high-byte facing, low-byte sub-id) tags rather than indices.
  3. Per-body kind → draw routine mapping. Needs Ghidra capture of the consumer (likely in the world_map_top overlay).
