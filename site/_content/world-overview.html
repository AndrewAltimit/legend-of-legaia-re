World overview work in progress

Each kingdom's world map (Drake, Sebucus, Karisto) populates a set of actor linked lists at PSX RAM 0x8007C34C..0x8007C36C. Each actor carries a world-space (X, Y, Z) position at offset +0x14. The data on this page is extracted from save-state captures of the top-view debug mode for each kingdom — see world-map subsystem for the controller, view-mode toggle, and per-frame state.

What you're looking at

Switch between the three kingdoms below. Each marker is one placed actor in the running game state — a town entrance, an NPC, a static tile, or a system object. Colour encodes the actor's owning list head:

  • list 0 (player / system) — head at 0x8007C34C; player actor + system markers.
  • list 1 (entities) — head at 0x8007C350; NPC entries, town gateways, encounter triggers.
  • list 2 (background) — head at 0x8007C354; static map decorations and tiles.
  • list 6 (extra) — head at 0x8007C36C; late-game / overflow placements (Karisto only in the captured snapshots).

World coordinates are signed; the placed range is roughly X ∈ [0, 16320], Z ∈ [0, 16320], with Y as elevation (negative is higher, PSX convention). The TMD that each actor renders isn't yet wired to the marker; once the actor → mesh resolver is reverse-engineered, the dots become textured 3D meshes in the same coordinates.

    What's next

    • Actor → TMD resolver. Each actor references a TMD via an intermediate animation/motion struct (pointer at actor+0x44 — not the TMD pointer table directly). Tracing that hop gives each marker its real mesh. (See memory note project_world_map_render_state.md.)
    • Rotation + scale. The actor struct holds these past offset +0x70 but the exact layout is per-subsystem; the renderer call FUN_80024C88(pos_vec, render_queue, list_head) takes the whole list and resolves per-actor transforms internally.
    • Texture-mapped 3D. Once the resolver lands, the existing software TMD rasteriser in the asset viewer can render the assembled scene textured.
