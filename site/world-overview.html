World overview work in progress

Each kingdom's overworld is built from a set of small flat TMDs (Z ∈ [-64, +64]) placed at world coordinates by the field VM. This page renders the world-map mesh data in real-time WebGL 3D straight from your disc image. World-map subsystem · scene-bundle format.

What's rendered, what's still WIP

In-game top-down view: upload a mednafen .mc[N] save state captured in the world-map debug top-view (slots 1 / 2 / 3 = Drake / Sebucus / Karisto for many of our recordings) and the page extracts the GPU framebuffer straight out of the save state's VRAM section. That framebuffer is what the game actually renders: the full kingdom continent composed of ~10,000 textured polygons per frame from the kingdom's TIM pack + tile-mesh data. No disc image required for this path - the save state already holds the rendered pixels.

Mesh inspector + placement scatter: if you load a disc image, you can also browse the per-kingdom TMD pack one mesh at a time (Mesh inspector mode) or see all decoded actor placements as a 3D scatter (Placement scatter mode). The placement walker mirrors FUN_8003A1E4's record reader and pulls Shift-JIS location names + (tmd_slot, x, z) coordinates from each kingdom's type-3 "Man" asset.

Placement table: the world-map actor positions are now extracted straight from each kingdom's bundle. The format is the type-3 "Man" asset (LZS-compressed slot 2 in the 7-asset table), parsed via FUN_8003A1E4's walker. Drake reports 43 world-placed / 50 records, Sebucus 18 / 23, Karisto 23 / 40 — the remainder are script-positioned (party members, story characters, etc.) and don't carry a static world coordinate.

Slot → TMD resolved: each placement's tmd_slot now cross-links to the exact mesh that fills it. The scene-pool TMDs come from slot 1 of the same 7-asset bundle (type byte 0x02) - an LZS-compressed pack with [u32 count, u32 word_offsets[count], TMD bodies] layout that the runtime walks via FUN_8001F05C case 2, calling FUN_80026b4c per body to fill positions 0..count-1 of the scene's TMD pool. The offsets are in 4-byte words, mirroring prot::timpack. Pack counts: Drake 40 TMDs / Sebucus 36 / Karisto 56; every scene-slot referenced by a placement record exists inside that pack (max slots: Drake 37, Sebucus 33, Karisto 54). Placements with tmd_slot ≥ 0xF0 address the global TMD pool (party-character overlays) instead - Drake's three party-on-the-world-map records use globals 0/1/2.

      How the placement data was extracted

      The page captures world-map state from top-view debug save states — the same view that FUN_801E76D4 enters when the debug toggle fires. Steps the extraction performs against each save state's main RAM:

      1. Read the active CDNAME label as ASCII at 0x80084548 and confirm it matches the expected kingdom (map01 / map02 / map03).
      2. Walk the seven linked-list heads at 0x8007C34C..0x8007C36C: each list starts at *head_ptr and chains via the actor's +0x00 next-pointer.
      3. For each actor, read (X, Y, Z) as three i16 at +0x14, flags at +0x10, and the actor's id byte at +0x50.
      4. Filter out zero-position entries (system actors that aren't placed in world-space).

      Results are baked into site/world-overview.json so the scatter view loads instantly without needing a save state at runtime.
