World overview work in progress

Each kingdom's overworld combines (a) a kingdom slot-1 TMD pack (40 / 36 / 56 meshes for Drake / Sebucus / Karisto) that holds both landmark meshes (castles, towers, bridges) and ground-tile meshes, and (b) a static MAN placement table that pins a subset of those meshes at world coordinates. The runtime additionally walks per-frame actor lists so the same pack supplies bulk terrain prims via a mode-switched per-prim renderer; the mechanism is now pinned (see world-map § bulk emit). This page renders the landmark layer in real-time WebGL 3D straight from your disc image — no save state required — and an optional toggle drops the unplaced slot-1 TMDs (the 35 / 30 / 39 meshes the MAN table doesn't pin) onto a canonical grid for visual classification.

What each mode needs and what it shows

Assembled scene (disc): with a disc image loaded, the page LZS-decodes the kingdom's 7-asset bundle — slot 0 (TIM_LIST) into a 1 MB software VRAM, slot 1 (TMD pack, [u32 count, u32 word_offsets[count], TMD bodies] walked via FUN_8001F05C case 2) into per-slot meshes. Every placement parsed from the type-3 "Man" asset is drawn at its world coordinates as a textured 3D mesh. Pack counts: Drake 40 TMDs / Sebucus 36 / Karisto 56. The MAN table only pins a handful of slots per kingdom (5 unique slots for Drake, 6 for Sebucus, 17 for Karisto); the rest are surfaced by the show unplaced slot-1 TMDs toggle below, laid out by class from site/world-overview/slot1_classification.toml (landmark / ground_tile / decoration / npc_token / unknown). For Drake the page also overlays 12 live-RAM-resolved actor placements baked into site/world-overview-live.json via scripts/pcsx-redux/resolve_actor_tmds.py — the runtime field-VM positions these via mesh-chain pointers at actor[+0x44], not from MAN. Distance-cue fog is applied by the runtime via FUN_80043390's overlay-mode dispatch table (see below); this WebGL view doesn't replicate the fog yet, so meshes render flat-colored.

Mesh inspector: with a disc loaded, browse the per-kingdom TMD pack one mesh at a time (orbit camera, auto-rotate).

Placement table: the world-map actor positions baked into site/world-overview.json are parsed from each kingdom's type-3 "Man" asset via FUN_8003A1E4's record walker. Drake reports 43 world-placed / 50 records, Sebucus 18 / 23, Karisto 23 / 40 — the remainder are script-positioned (party members, story characters, etc.) and don't carry a static world coordinate. Each placement's tmd_slot resolves into the kingdom pack (max slot Drake 37, Sebucus 33, Karisto 54); placements with tmd_slot ≥ 0xF0 address the global TMD pool (party-character overlays) instead. Drake additionally surfaces 12 live-RAM actor placements via scripts/pcsx-redux/resolve_actor_tmds.py: each world-map actor's mesh_head[+0x44] chain is walked to find the containing TMD via backward magic-word search, then mapped to a slot index. The result is baked into site/world-overview-live.json; the viewer overlays those placements on top of the MAN entities so the bridge towers, gates, small castle, and other actor-positioned features render correctly.

Bulk continent terrain (mechanism pinned). The ~4300 POLY_FT4 prims that tile the kingdom continent in the dev-menu top-view come out of ordinary case-5 TMD rendering whose per-prim dispatch is mode-switched to overlay-resident renderers when the world-map overlay is paged in. FUN_80043390 selects one of two function-pointer tables based on _DAT_1F800394 & 1 — SCUS at 0x8007657C when clear, world-map overlay at 0x801F8968 when set. The overlay table's eight high-mode renderers at 0x801F7644..0x801F8690 are the SCUS sibling bodies plus a per-vertex distance-cue fog post-process (GTE dpcs / dpct + a per-Z RGB LUT tint). The source mesh data is the same kingdom slot-1 TMD pack the landmarks draw from; the runtime just routes prims through the fog-enabled leaves when rendering the world map. Full mechanism + per-slot fog delta at world-map § bulk emit; CLI at mednafen-state prim-dispatch-table <save>.

Slot 4 of the kingdom bundle. Slot 4 (type byte 0x05, ~32 KB) was previously hypothesized to be the dev-menu top-view wireframe / coastline data; visual inspection falsified that reading across every projection and topology mode. The container is solved (15 / 16 / 16 sub-bodies, byte-verified vs live RAM at 0x8011A624) but the records' semantic remains open. Dynamic memory-watchpoint capture against the dev-menu top-view registered zero reads, so the consumer runs only at scene-load (or via input the captured state doesn't trigger). Slot 4 is not the bulk continent terrain source — that puzzle is now answered by FUN_80043390's dispatch table, above. See slot-4 records doc.

Ocean tile + wave animation. The world-map ocean is a static 4bpp tile + per-frame CLUT cycling, both shipped on disc inside slot 0 (TIM_LIST) of the same kingdom bundle. The texture sits at VRAM (768, 256) 64x256 halfwords with its base CLUT at (0, 506) 256x1; the wave-ramp data fills the top-left 96x96 logical pixels of the texture page (the rest is shared with other tiles in 4bpp mode). The animation is 13 frames x 16 BGR555 entries wrapped in 532-byte "CLUT-only TIM" records at TIM_LIST slots 3-5 (Sebucus / Karisto) or 10-15 (Drake); the runtime swaps one frame at a time onto the first 16 entries of the CLUT, cycling a bright-blue wave peak through indices 0-7. The animation table is byte-identical across all three kingdoms (verified by SHA-256, disc-gated crates/web-viewer/tests/ocean_assets.rs). The 3D viewer extracts these via legaia_web_viewer::ocean::find_ocean_assets at scene load and renders the ocean as a textured plane with per-frame CLUT swap in the fragment shader. Before a disc is loaded the renderer falls back to a solid kingdom tint (baked into world-overview.json by scripts/mednafen/resolve_bulk_terrain.py's pick_ocean_color).

      World-map quick-travel menu (whole Legaia)

      The PSX retail build ships a 16-entry landmark name table at DAT_80073B18 + a 20-record placement table at DAT_80073A98 in SCUS_942.54 — the data the in-game world-map menu uses to draw labelled markers when the player presses the "visit" button. Each placement record is 6 bytes: (name_idx, discovery_flag, scene_id, menu_x, menu_y). The walker (FUN_80030628 case 0x19) skips records whose discovery flag (offset +0x20 in the fourth flag bank) is still cleared, and dedupes consecutive entries that share a name — so 16 landmarks expose 20 entry-point records with 3 of them flagged with multiple sub-entries (Biron Monastery, Drake Castle, Rim Elm, Ratayu have 2-3 each).

      Coordinates are the in-game menu screen-space (menu_x ∈ 22..102, menu_y ∈ 13..108). Hover a dot for the name and the destination scene id. Data is parsed from SCUS_942.54 at disc-load time; load a disc above to populate.

       

      Landmarks (16)

        How the placement data was extracted

        The page captures world-map state from top-view debug save states — the same view that FUN_801E76D4 enters when the debug toggle fires. Steps the extraction performs against each save state's main RAM:

        1. Read the active CDNAME label as ASCII at 0x80084548 and confirm it matches the expected kingdom (map01 / map02 / map03).
        2. Walk the seven linked-list heads at 0x8007C34C..0x8007C36C: each list starts at *head_ptr and chains via the actor's +0x00 next-pointer.
        3. For each actor, read (X, Y, Z) as three i16 at +0x14, flags at +0x10, and the actor's id byte at +0x50.
        4. Filter out zero-position entries (system actors that aren't placed in world-space).

        Results are baked into site/world-overview.json so the scatter view loads instantly without needing a save state at runtime.
