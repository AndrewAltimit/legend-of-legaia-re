World overview work in progress

Each kingdom's overworld combines (a) a continent terrain mesh built at runtime as ~10,000 textured polygons in a GPU primitive pool and (b) a sparse set of landmark TMDs (towns, dungeons, NPCs) placed at world coordinates from the kingdom's bundle. This page renders both in real-time WebGL 3D — the landmark TMDs straight from your disc image; the continent terrain (currently) from a top-view mednafen save state, until the continent generator is fully reversed. World-map subsystem · scene-bundle format.

What each mode needs and what it shows

Assembled scene (disc): with a disc image loaded, the page LZS-decodes the kingdom's 7-asset bundle — slot 0 (TIM_LIST) into a 1 MB software VRAM, slot 1 (TMD pack, [u32 count, u32 word_offsets[count], TMD bodies] walked via FUN_8001F05C case 2) into per-slot meshes. Every placement parsed from the type-3 "Man" asset is drawn at its world coordinates as a textured 3D mesh. Pack counts: Drake 40 TMDs / Sebucus 36 / Karisto 56. This renders the landmark layer (~43 buildings/dungeons/NPCs for Drake) but not the continent terrain beneath it — the MAN-asset placement table only references entity TMDs (player/NPC instances at slots 0+35), leaving 38 of Drake's 40 TMDs un-placed. Those un-placed TMDs include real terrain chunks but their world positions live in a structure we haven't yet pinned (likely set by runtime field-VM script at scene init, not in any static slot of the bundle). Each kingdom's PROT block also carries a SECOND 7-asset table at +8/+9 (Drake PROT 0093 = 70 TMDs) which holds sub-scenes (battle arena, town interiors), not world-map continent; exposed via continent_pack_* WASM accessors for inspection.

Framebuffer (save state): until the continent generator is reversed, the only way to see the full overworld is to upload a mednafen .mc[N] save state captured while the world-map debug top-view is active. The page then extracts either the rendered GPU framebuffer straight out of VRAM (fast, pixel-accurate, fixed 320×240) or replays the live primitive pool (~10,000 textured POLY_FT4 packets, pan/zoom in WebGL2). No disc image required for this path. Tracking the generator RE in world-map subsystem.

Mesh inspector: with a disc loaded, browse the per-kingdom TMD pack one mesh at a time (orbit camera, auto-rotate).

Placement table: the world-map actor positions baked into site/world-overview.json are parsed from each kingdom's type-3 "Man" asset via FUN_8003A1E4's record walker. Drake reports 43 world-placed / 50 records, Sebucus 18 / 23, Karisto 23 / 40 — the remainder are script-positioned (party members, story characters, etc.) and don't carry a static world coordinate. Each placement's tmd_slot resolves into the kingdom pack (max slot Drake 37, Sebucus 33, Karisto 54); placements with tmd_slot ≥ 0xF0 address the global TMD pool (party-character overlays) instead.

      World-map quick-travel menu (whole Legaia)

      The PSX retail build ships a 16-entry landmark name table at DAT_80073B18 + a 20-record placement table at DAT_80073A98 in SCUS_942.54 — the data the in-game world-map menu uses to draw labelled markers when the player presses the "visit" button. Each placement record is 6 bytes: (name_idx, discovery_flag, scene_id, menu_x, menu_y). The walker (FUN_80030628 case 0x19) skips records whose discovery flag (offset +0x20 in the fourth flag bank) is still cleared, and dedupes consecutive entries that share a name — so 16 landmarks expose 20 entry-point records with 3 of them flagged with multiple sub-entries (Biron Monastery, Drake Castle, Rim Elm, Ratayu have 2-3 each).

      Coordinates are the in-game menu screen-space (menu_x ∈ 22..102, menu_y ∈ 13..108). Hover a dot for the name and the destination scene id. Data is parsed from SCUS_942.54 at disc-load time; load a disc above to populate.

        How the placement data was extracted

        The page captures world-map state from top-view debug save states — the same view that FUN_801E76D4 enters when the debug toggle fires. Steps the extraction performs against each save state's main RAM:

        1. Read the active CDNAME label as ASCII at 0x80084548 and confirm it matches the expected kingdom (map01 / map02 / map03).
        2. Walk the seven linked-list heads at 0x8007C34C..0x8007C36C: each list starts at *head_ptr and chains via the actor's +0x00 next-pointer.
        3. For each actor, read (X, Y, Z) as three i16 at +0x14, flags at +0x10, and the actor's id byte at +0x50.
        4. Filter out zero-position entries (system actors that aren't placed in world-space).

        Results are baked into site/world-overview.json so the scatter view loads instantly without needing a save state at runtime.
